# 05-design-system.md — The EasyLab Club

## 1. Purpose

Define a clear, minimal, and consistent visual system.

This document removes ambiguity for AI generation.

---

## 2. Design Principles

* clarity over aesthetics
* structure over decoration
* hierarchy is always visible
* minimal cognitive load
* no visual noise

Avoid:

* startup flashy UI
* luxury/editorial styles
* over-designed components

---

## 3. Visual Tone

The UI must feel:

* clean
* structured
* modern
* operational

Not:

* corporate heavy
* playful
* experimental

---

## 4. Color System

Primary:

* Blue → action / primary CTA

Secondary:

* Orange → accent / highlight

Neutral:

* White → background
* Dark slate → text

Rules:

* max 2 accent colors per screen
* no gradients unless very subtle
* avoid color overload

---

## 5. Typography

Font:

* Sans-serif (modern, readable)

Hierarchy:

* H1 → bold, large, high impact
* H2 → strong, structured
* H3 → section titles
* Body → clear, readable

Rules:

* no decorative fonts
* no excessive font mixing

---

## 6. Layout System

Structure:

* max-width container
* consistent spacing between sections
* vertical rhythm is critical

Sections:

* always clearly separated
* no overlapping content
* no complex compositions

---

## 7. Components Rules

All components must be:

* reusable
* predictable
* simple

No:

* one-off custom elements
* experimental layouts

---

## 8. Buttons & CTA

Primary CTA:

* solid background
* high contrast

Secondary CTA:

* outline or minimal

Rules:

* max 2 CTAs per section
* clear action labels

---

## 9. Cards

Used for:

* Lab modules
* Programs
* Features

Structure:

* title
* short description
* optional price

Rules:

* consistent spacing
* no visual overload

---

## 10. Spacing System

Use consistent spacing scale:

* small
* medium
* large

Rules:

* avoid random spacing
* maintain vertical rhythm

---

## 11. Network Section

Must feel:

* subtle
* aspirational
* secondary

Avoid:

* community-style UI
* social platform patterns

---

## 12. Critical Rule

The design system must reduce interpretation.

If a component requires explanation:
→ it is too complex

---

## 13. Output Expectation

After applying this system:

* UI is consistent
* structure is clear
* no design drift

---
